# Web Security Scenario Lab

Статический QA- и regression-стенд для browser-extension детекторов, анализа защитного рендеринга и демонстраций security research в контролируемых средах.

> Проект не предназначен для практического злоупотребления, фишинга, мошенничества или иной вредоносной деятельности. Внутри сценариев могут встречаться случайно сгенерированные ссылки, вводящие в заблуждение подписи, визуально скрытые элементы и другие имитации потенциально опасных техник.

## Позиционирование

`Web Security Scenario Lab` — это отдельный local-first проект для генерации воспроизводимых client-side веб-сценариев, предназначенных для валидации browser detectors и смежных browser-security инструментов.

Основные фокусы:

- browser-extension QA
- regression testing
- security research demos
- red team validation в контролируемых средах
- blue team validation в контролируемых средах
- purple team validation в контролируемых средах

В отличие от платформ для симуляции фишинговых кампаний или end-user awareness демонстраций, данный стенд сосредоточен на воспроизводимых client-side паттернах веб-страниц для проверки browser detectors на скрытые или обманные техники рендеринга.

## What This Is Not

Этот проект не является:

- платформой для фишинговых кампаний
- LMS-системой или учебным порталом для пользовательской awareness-подготовки
- набором end-user awareness демо для нетехнической аудитории
- сервисом URL reputation
- crawler-based сканером интернета
- системой live-сбора или массового веб-скрейпинга
- источником доменной репутации или внешней threat intelligence лентой
- фермой браузерной автоматизации для массового зондирования интернета
- заменой для реального threat intel, SOC telemetry или production blocklists

## Модель покрытия модулей

Модули делятся на две группы.

Реализуются в первую очередь:

- `visual-manipulation`
- `link-domain-security`

Planned modules с документированными placeholder-страницами с самого начала:

- `trigger-phrases` - planned module
- `prompt-splitting` - planned module
- `api-interception` - planned module

Planned modules включаются в проект для сохранения стабильной информационной архитектуры, навигации и точек расширения на будущее. Их наличие не должно создавать ощущение, что половина проекта пустая. Это forward-compatible заглушки под будущие scenario packs.

## Примечание по вкладу

Planned modules должны присутствовать в структуре проекта как документированные placeholder-разделы. Если кто-то захочет помочь, он сможет присылать pull request'ы с манифестами сценариев, улучшениями заглушек, evaluation metadata или полной реализацией сценариев для этих модулей.

## Базовые требования

- Только статический проект: `HTML`, `CSS`, `JavaScript`, `JSON`, ассеты.
- Без backend, базы данных, SSR и серверной бизнес-логики.
- Локальная разработка должна одинаково работать на Windows и Linux.
- Стенд должен открываться в современных браузерах: Chrome, Edge, Firefox, Safari и других Chromium-браузерах.
- С первого этапа нужны `RU/EN`.
- Нужна главная страница с переходами к страницам модулей.
- Нужны не только фиксированные образцы, но и динамическая генерация вариантов.
- Весь runtime должен выполняться в браузере, чтобы не нагружать сервер.
- При деплое должна быть минимальная поверхность атаки: сервер только раздаёт файлы.
- Сценарии должны рассматриваться как benchmark-lite активы для проверки детекторов, а не просто как набор демо-страниц.

## Рекомендуемый стек

- `HTML5`
- `CSS3`
- `Vanilla JavaScript` на `ES modules`
- `JSON`-манифесты сценариев и evaluation metadata
- `Vite` для локальной разработки и сборки статики
- `Playwright` для локальных и CI-проверок

`Vite` и `Playwright` входят только в контур разработки и тестирования. В продакшн-деплой они не обязательны.

## Общая структура

- `index.html`: главная страница с каталогом модулей и сценариев.
- `pages/visual-manipulation/`: сценарии по скрытому контенту и CSS-обфускации.
- `pages/link-domain-security/`: сценарии по ссылкам, доменам и гомографам.
- `pages/trigger-phrases/`: placeholder-страницы planned module.
- `pages/prompt-splitting/`: placeholder-страницы planned module.
- `pages/api-interception/`: placeholder-страницы planned module.
- `pages/shared/`: общие layout-компоненты и demo-helper'ы.
- `assets/`: шрифты, локальные JS-библиотеки, изображения, образцы фонов.
- `data/scenarios/`: JSON-манифесты и наборы конфигураций для вариативности.
- `data/evaluation/`: expected outcomes, severity, tags и rationale metadata.
- `src/`: движок сценариев, i18n, seeded random, evaluation layer, bootstrap и benchmark-lite UI.

## Функциональные принципы

- Каждая страница сценария может работать в fixed и randomized режимах.
- Случайность должна быть воспроизводимой через `seed`.
- Query parameters должны позволять делать deep link, например:
  - `?lang=ru`
  - `?seed=1042`
  - `?module=visual-manipulation`
  - `?scenario=hidden-text-mixed`
- Динамические изменения содержимого должны происходить только в браузере.
- Подозрительный контент должен генерироваться только из внутренних шаблонов, а не из пользовательского ввода.
- Planned modules должны присутствовать в навигации как placeholder-элементы с видимым статусом.
- Сценарии должны отдавать evaluation outputs, а не только визуальную вариативность.

## Evaluation Outputs

Каждый зрелый сценарий должен уметь отдавать исследователю такие артефакты, как:

- expected signal
- detector should fire / should not fire
- severity
- tags
- объяснение, почему страница подозрительна или намеренно benign
- coverage dimensions, встроенные в данный сценарий

Долгосрочная цель состоит в том, чтобы превратить стенд в benchmark-lite, а не в разрозненный набор страниц. Поэтому `why flagged` panel и coverage matrix стоит рассматривать как части целевой архитектуры уже сейчас.
