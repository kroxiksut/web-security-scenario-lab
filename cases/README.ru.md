# Реестр кейсов

В этом каталоге хранятся исследовательские кейсы, полученные из реальных наблюдений, контролируемых reference-материалов, архивных примеров и обезвреженных воспроизведений.

Назначение `cases/` — связать реальные находки и сценарии стенда.

## Зачем это нужно

Основной scenario lab содержит контролируемые воспроизводимые страницы. Реестр `cases/` нужен для того, чтобы исследователь мог фиксировать происхождение идей, отмечать сигналы, замеченные в реальных условиях, и понимать, были ли эти наблюдения уже перенесены в benchmark-lite сценарии.

## Основные цели

- сохранять исследовательские reference-материалы, не превращая проект в live threat feed
- связывать реальные находки с локальными воспроизводимыми сценариями
- документировать значимые для детекторов сигналы в структурированном виде
- поддерживать browser-extension QA и regression-проверки реальными примерами
- держать записи безопасными для публикации, пригодными для review и редактируемыми при необходимости редактирования

## Источник истины

Кейсы нужно хранить в Markdown-файлах.

Причины:

- удобнее diff и review в Git
- безопаснее, чем использовать активный HTML как основной формат
- проще редактировать опасные ссылки и нежелательный контент
- проще в будущем расширять структурированной metadata
- проще генерировать HTML или JSON поверх Markdown, чем наоборот

HTML может появиться позднее как производное представление, но основным форматом хранения и review остаётся Markdown.

## Структура каталога

```text
cases/
|-- README.md
|-- README.ru.md
|-- CASE-TEMPLATE.md
|-- CASE-TEMPLATE.ru.md
|-- INDEX.md
|-- visual-manipulation/
|-- link-domain-security/
|-- trigger-phrases/
|-- prompt-splitting/
|-- api-interception/
|-- cross-module/
`-- assets/
```

## Правила категорий

Новая категория создаётся, если выполняется хотя бы одно условие:

- кейс явно относится к одному модулю-детектору
- кейс относится к устойчивому семейству сигналов, которое может переиспользоваться в нескольких сценариях
- кейс затрагивает несколько модулей и ему нужен нейтральный раздел

Не нужно создавать отдельную категорию под каждую микроскопическую эвристику.

Предпочтительно использовать:

- модульные категории для detector-specific находок
- `cross-module/` для кейсов, сочетающих несколько семейств сигналов
- `assets/` для скриншотов, sanitized HTML, архивов и вспомогательных материалов

## Правила именования

Каждый кейс должен иметь стабильный идентификатор.

Рекомендуемый шаблон:

- `VM-YYYY-NNN.md` для `visual-manipulation`
- `LDS-YYYY-NNN.md` для `link-domain-security`
- `TP-YYYY-NNN.md` для `trigger-phrases`
- `PS-YYYY-NNN.md` для `prompt-splitting`
- `API-YYYY-NNN.md` для `api-interception`
- `XM-YYYY-NNN.md` для `cross-module`

Пример:

- `LDS-2026-001.md`

## Правила безопасности

Реестр кейсов не должен превращаться в публичный список опасных активных ссылок.

- при необходимости использовать redacted URL
- для отображения использовать безопасный вид вроде `example[.]com`
- не публиковать токены, credentials, персональные данные и live malicious payloads
- не превращать кейсы в исполняемые демо
- фиксировать только ту информацию, которая нужна для исследования, валидации и маппинга на сценарии

## Рекомендуемые статусы

Подходящие статусы кейсов:

- `draft`
- `confirmed`
- `redacted`
- `archived`
- `mapped-to-lab`
- `needs-reproduction`

## Маппинг на сценарии

Хороший кейс должен отвечать на вопрос, был ли он перенесён в lab.

Рекомендуемые поля:

- `reproducibleInLab`
- `scenarioMapping`
- `coverageDimensions`
- `expectedSignal`

## Переводы от сообщества

Кейсы по умолчанию ведутся на английском языке.

Документация и process-файлы сейчас могут существовать на английском и русском, а в будущем расширяться community-переводами через дополнительные файлы `*.xx.md`.
