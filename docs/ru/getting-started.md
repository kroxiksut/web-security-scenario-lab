# Запуск

Английское зеркало: [`../en/getting-started.md`](../en/getting-started.md).

Стенд — **статический сайт, работающий только на стороне клиента**: без бэкенда, без базы данных, без
серверного рендеринга. Для разработки используется Vite; результат сборки — обычные HTML/CSS/JS/JSON,
которые раздаст любой примитивный файловый сервер.

Ниже два независимых руководства. Выберите своё и идите по нему сверху вниз — переключаться между
ними не нужно:

- [**Windows**](#windows) — PowerShell
- [**Linux**](#linux) — bash

Общие для обеих систем разделы — [параметры глубоких ссылок](#параметры-глубоких-ссылок) и
[проверки качества](#проверки-качества) — идут в конце.

## Требования

| Требование | Примечание |
| --- | --- |
| **Node.js `^20.19` / `^22.12` / `>=24`** | Требуется Vite 8 (`^20.19.0 \|\| >=22.12.0`) и Vitest 4. Рекомендуется Node 22 LTS или 24. |
| **npm 10+** | Поставляется вместе с поддерживаемыми версиями Node. |
| Современный браузер | Chrome, Edge, Firefox, Safari или любой браузер на Chromium. |
| Git | Нужен только чтобы клонировать репозиторий. |

Больше ничего не требуется — ни Python, ни Docker, ни отдельный веб-сервер.

## Windows

Всё выполняется в **PowerShell**. Откройте меню «Пуск» → наберите `PowerShell` → запустите
**Windows PowerShell** (или Windows Terminal). Права администратора не нужны.

### 1. Проверьте Node

```powershell
node -v
npm -v
```

Если `node -v` показывает версию ниже `v20.19` или команда не найдена — поставьте Node 22 LTS с
[nodejs.org](https://nodejs.org/) и откройте новое окно PowerShell.

### 2. Перейдите в папку с проектами и склонируйте репозиторий

```powershell
cd $HOME\Projects
git clone <адрес-репозитория>
cd web-security-scenario-lab
```

Если папки `Projects` нет, создайте её: `New-Item -ItemType Directory $HOME\Projects`. Репозиторий уже
склонирован? Просто перейдите в его папку:

```powershell
cd C:\путь\к\web-security-scenario-lab
```

> **Синхронизируемые папки.** Если проект лежит в OneDrive, Яндекс.Диске или Dropbox, приостановите
> синхронизацию перед следующим шагом: клиенты синхронизации блокируют файлы внутри `node_modules` и
> могут уронить установку или оставить дерево записанным наполовину.

### 3. Установите зависимости

```powershell
npm run setup
```

Эта команда проверит версии Node и npm, предупредит, если репозиторий лежит в синхронизируемой папке,
и выполнит обе установки. Есть и эквивалентная обёртка на PowerShell — она ничего не делает сама,
только вызывает тот же скрипт:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\setup.ps1
```

Флаг `-ExecutionPolicy Bypass` нужен потому, что PowerShell по умолчанию не запускает неподписанные
скрипты. Если возиться с этим не хочется — используйте `npm run setup`, результат тот же.

Проверить окружение, ничего не устанавливая: `npm run setup -- --check`.

Обе команды делают одно и то же:

```powershell
npm install
npm run frameworks:install
```

Вторая нужна один раз на свежем клоне: сценарии React 17 и React 19 ставятся во **вложенные
`node_modules`**, чтобы каждый `react-dom` лежал рядом со своей версией `react`, и корневой
`npm install` их не захватывает. Пропустите её (`npm run setup -- --skip-frameworks`) — сломаются
только страницы React 17 / React 19, остальной стенд соберётся и заработает. Подробности — в
[Добавлении фреймворка или версии](./adding-frameworks.md).

### 4. Запустите стенд

```powershell
npm run dev
```

Vite выведет адрес — по умолчанию `http://localhost:5173/`. Откройте его в браузере: это главная
страница с карточками модулей. Горячая перезагрузка включена, TypeScript компилируется на лету.

Остановить сервер — `Ctrl+C` в том же окне PowerShell.

Другой порт или доступ по локальной сети:

```powershell
npm run dev -- --port 5180
npm run dev -- --host
```

### 5. Соберите статику и раздайте её (по необходимости)

Этот шаг нужен, только если вы проверяете стенд так, как он будет развёрнут.

```powershell
npm run build
npm run preview
```

`npm run build` кладёт результат в `dist\`, `npm run preview` раздаёт его на
`http://localhost:4173/`.

Проверить, что стенд работает на любом примитивном статическом сервере (в репозитории есть такой,
без единой зависимости):

```powershell
node serve.mjs dist
node serve.mjs dist --port 8080
$env:PORT = 8080; node serve.mjs dist
```

Подойдёт и любой другой:

```powershell
npx serve dist
py -3 -m http.server 4173 --directory dist
```

**Раздавать нужно `dist\`, а не корень репозитория**: исходные страницы ссылаются на `src\main.ts`,
который умеет разрешить только сборщик, — обычный файловый сервер отдал бы браузеру сырой TypeScript.

Открывать `dist\index.html` двойным щелчком (`file://`) — **приятный бонус, а не поддерживаемый
режим**: модульные скрипты и `fetch()` локальных JSON-манифестов подчиняются правилам origin, которые
в разных браузерах ведут себя по-разному. Используйте статический сервер.

## Параметры глубоких ссылок

Каждая страница принимает query-параметры, поэтому конкретное состояние сценария можно передать
коллеге или зашить в скрипт:

| Параметр | Значения | Действие |
| --- | --- | --- |
| `?seed=` | любая строка, например `1042` | Сид для PRNG: рандомизированный сценарий воспроизводится точь-в-точь. По умолчанию `default`. |
| `?focus=1` | `1` / `0` | Изолированный тестовый режим — прячет UI оболочки (меню, боковые панели), оставляя только DOM сценария. |
| `?lang=` | `en` / `ru` | Принудительно задаёт язык интерфейса, игнорируя сохранённый выбор. |
| `?theme=` | `light` / `dark` / `auto` | Принудительно задаёт цветовую тему. |

Параметры комбинируются:
`http://localhost:5173/pages/visual-manipulation/hidden-text.html?seed=1042&focus=1&lang=ru`

Учтите: они переключают **оболочку стенда**, а не полезную нагрузку — сценарии, которые размещают
двуязычный контент, размещают его независимо от языка интерфейса.

## Проверки качества

Перед тем как предлагать изменения, должны проходить все четыре проверки. Одной командой (одинаково в
обеих системах):

```bash
npm run verify
```

Она последовательно выполняет:

```bash
npm run typecheck
npm run lint
npm run test
npm run build
```

Тесты покрывают **только движок и инфраструктуру** (сид-PRNG, загрузка манифестов, валидация по
JSON-схеме, i18n, разрешение evaluation). Страницы сценариев намеренно вариативны и несовершенны и
тестами не покрываются — см. [Contributing](../../CONTRIBUTING.md).

Форматирование доступно как `npm run format` / `npm run format:check`. Несколько унаследованных
файлов заведомо не соответствуют формату; репозиторий не гейтится на `format:check`.
