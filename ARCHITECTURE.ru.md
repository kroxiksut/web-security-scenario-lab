# Архитектура

## Цель

Создать статический browser-security scenario lab для browser-extension QA, regression testing, controlled security research demos и red/blue/purple team validation без backend-сложности и без увеличения серверной поверхности атаки.

## Модель позиционирования

Стенд должен проектироваться прежде всего как среда валидации детекторов.

- основной use case: проверка browser-extension детекторов на скрытое или обманное поведение веб-страниц
- вторичный use case: воспроизводимые security research demos в контролируемых средах
- архитектурный вектор: benchmark-lite валидация, а не универсальная security-simulation платформа
- принцип разграничения: не phishing platform, не crawler, не reputation service

## Архитектурная модель

- модель поставки: только статический хостинг
- модель runtime: только клиентская логика
- модель разработки: local-first, кроссплатформенно
- браузерная модель: standards-first, без зависимости от API расширений
- модель данных: манифесты сценариев плюс seed-управляемые мутации
- модель модулей: каждый модуль детекции содержит настоящие сценарии; добавление модуля не должно менять информационную архитектуру
- модель оценки: каждый зрелый сценарий должен отдавать expected outcomes и rationale metadata
- модель case-registry: реальные reference-наблюдения хранятся как Markdown-кейсы и по возможности маппятся на сценарии lab

## Навигационная модель

- `index.html` является главной страницей стенда.
- Главная страница содержит:
  - описание проекта
  - карточки модулей, сгруппированные по тому, на кого нацелена манипуляция (человек-читатель /
    AI-агент, читающий страницу)
  - карточки сквозных представлений: каталог сценариев и матрица покрытия фреймворков
  - переключатель языка
  - переходы к группам сценариев
- Оболочка (`src/shell/mountShell.ts`) достраивает хедер, меню модулей, сайдпанель и футер вокруг
  собственного содержимого страницы, поэтому страница сценария несёт только свой DOM. Статусных
  меток у модулей в навигации нет: у каждого есть настоящие сценарии.
- Каждая страница модуля содержит:
  - обзор модуля
  - список сценариев со ссылками на их страницы
  - ссылку назад на главную страницу
- Каждая страница сценария содержит:
  - поле seed, кнопки reroll, reset и копирования deep link
  - панель оценки, отрисованную из манифеста сценария
  - focus-режим (`?focus=1`), убирающий оболочку и оставляющий только DOM сценария

## Модель статусов модулей

Модули детекции, соответствующие 1:1 модулям детектора. Все пять содержат настоящие сценарии.

Нацелены на человека-читателя:

- `visual-manipulation` — 22 сценария (3 основные страницы, benign-контроль и 18 фреймворк-версионных вариантов)
- `link-domain-security` — 2 сценария

Нацелены на AI-агента, читающего страницу:

- `trigger-phrases` — 3 сценария
- `prompt-splitting` — 5 сценариев
- `api-interception` — 4 сценария

**Контракт benign-сценариев:** каждый модуль содержит и позитивные сценарии (`shouldFire: true`,
требуется `whyFlagged`), и минимум один benign-контроль ложных срабатываний (`shouldFire: false`,
требуется `whyBenign`). Схема манифеста следит за тем, какое именно обоснование обязательно. По стенду
это 29 позитивных сценариев и 7 benign-контролей. Benign-контроль пишется так, чтобы задевать
*наивную* версию эвристики собственного модуля: пара задаёт границу решения, а не только её
позитивную сторону.

Добавление модуля, сценария или версии фреймворка не должно требовать изменения информационной
архитектуры.

## Движок сценариев

Движок сценариев остаётся простым и детерминированным.

- страница объявляет себя через data-атрибуты body (`data-scenario`, `data-module`, `data-root`);
  движок по id подгружает соответствующие манифест и модуль поведения
- манифесты — локальный JSON, проверяемый по JSON-схеме (Ajv) на этапе тестов и сборки
- поведение сценария — типизированный TypeScript в `src/scenarios/<id>.ts`, экспортирующий `run(ctx)`
  с сид-генератором `rng` и корневым элементом `root`; общие драйверы лежат в `src/scenarios/_shared/`
- случайность даёт сид-PRNG (mulberry32), сид берётся из `?seed=`; `Math.random` не используется
- содержимое рендерится только из внутренних шаблонов, никогда из пользовательского ввода
- содержимое меняется при первом открытии, при ручном reroll (новый сид) и по внутристраничным
  таймерам или взаимодействию
- метаданные сценария рисуются в панели оценки прямо из манифеста, поэтому UI страницы и ground truth
  не могут разойтись

## Evaluation Layer

Стенд не должен останавливаться на генерации страниц. Он должен также описывать ожидаемое поведение детектора.

Каждый зрелый сценарий должен уметь публиковать:

- expected signal
- detector should fire или should not fire
- severity
- tags
- объяснение, почему страница подозрительна или намеренно benign
- coverage dimensions, например hidden text, tiny font, off-screen placement, z-index masking, misleading anchor text, punycode или mixed-script паттерны, protocol abuse

## Benchmark-Lite UI

Именно это переводит проект из состояния просто набора страниц в benchmark-lite regression lab.
Уже сделано:

- панель `why flagged` / `why benign`, отрисованная из манифеста активного сценария
- каталог сценариев: все сценарии по модулям с ожидаемым сигналом, вердиктом, severity и тегами
- матрица покрытия фреймворков, строящаяся из `data/frameworks.json`
- scenario evaluation summary, привязанная к активному seed и варианту страницы
- явное разделение между intentionally suspicious и intentionally benign страницами

Пока целевое состояние:

- отрисовка `detectionExamples` рядом со сценарием как обучающего материала (поле манифеста
  зарезервировано и заполняется, рендеринг отложен)
- представление покрытия по эвристикам / coverage dimensions, а не по модулям и фреймворкам
- машиночитаемый экспорт ground truth для автоматической оценки детектора

## Покрытие библиотек

Стенд должен включать страницы, имитирующие типовые frontend-окружения:

- обычный DOM
- `jQuery`
- `React`
- `Vue`
- `Svelte`
- `Angular`
- `Solid`
- `Preact`
- `Lit` / Web Components
- `htmx`, `Alpine`
- `Tailwind` (поверхность CSS-обфускации через utility-классы)
- `Shadow DOM`
- встраивание через `iframe`

Цель здесь в устойчивости детекторов, а не в полном повторении возможностей фреймворков. Каждая такая страница должна оставаться минимальной и сосредоточенной на различиях поведения DOM.

**Набор библиотек/фреймворков намеренно открыт и со временем может расширяться** по мере
развития реального веба; добавление библиотеки не должно требовать смены информационной
архитектуры. Политика версий: последний стабильный патч, мажор фреймворка `>= 2022`, с
осознанными legacy-but-ubiquitous исключениями (например, jQuery 3.7.x), поскольку детектор
обязан работать и по старым страницам. Все библиотеки вендорятся/пинятся локально — без
runtime-CDN.

### Мульти-версионное покрытие

**Версия библиотеки — самостоятельное измерение покрытия**: разные мажоры дают разный
DOM/рантайм-след (делегирование событий, маркеры гидрации, атрибуты фреймворка), который
детектор обязан пережить. Несколько версий уживаются через **npm-алиасы**
(`react18@npm:react@18`, `react19@npm:react@19`), по одной версии на страницу сценария, в
версионированных папках (`frameworks/react/v18/`) и описываются матрицей `data/frameworks.json`
(библиотека → версии → `{alias, releaseDate, whyIncluded}`). Версии пинятся точно и намеренно
не обновляются — легаси-покрытие и есть цель. Матрица расширяема по двум осям (новые библиотеки
и новые версии). Строится не сразу целиком: сначала репрезентативные мажоры (React, Vue,
jQuery), шире — позже.

## Язык и инструментарий

- **Движок сценариев и общая инфраструктура** пишутся на `TypeScript` там, где это повышает
  надёжность и поддерживаемость (сид-PRNG, загрузчик/валидатор манифестов, i18n, резолвер
  evaluation).
- **Разметка и рантайм-DOM сценариев остаются намеренно «грязными»** — несовершенный HTML, странные
  события и мутации по таймеру повторяют реальные вредоносные страницы, и «причёсывать» их нельзя.
  При этом само поведение пишется как типизированный TypeScript в `src/scenarios/`: это удерживает
  авторство поддерживаемым, ничуть не делая итоговый DOM аккуратнее.
- **Политика зависимостей движка: стремимся к нулю runtime-зависимостей.** Оболочка, i18n,
  маршрутизация, сид-PRNG и настройки написаны вручную поверх DOM API; движок не использует UI-фреймворк,
  поэтому страницы сценариев остаются нейтральным DOM для проверяемого детектора. `Ajv` — единственная
  санкционированная библиотека, и работает она на этапе сборки/тестов, а не в рантайме.
- Инструменты сборки/тестов: `Vite` (сборка), `Vitest` (юнит-тесты движка), `Ajv` (валидация
  схемы манифестов), `ESLint` + `Prettier`. `Playwright` — опционально/вручную только для E2E
  расширения.
- Тулчейну требуется Node.js `^20.19` / `^22.12` / `>=24`; см.
  [`docs/ru/getting-started.md`](./docs/ru/getting-started.md).

## Поддержка динамических страниц

Динамические страницы входят в обязательный объём.

- отложенная вставка DOM-узлов
- периодическая замена содержимого
- изменение атрибутов
- перестановка class names
- перегенерация ссылок
- смена фонов и контрастов
- изменение таймингов overlay

Все такие изменения должны выполняться только в браузере.

## Правила разработки

- использовать только кроссплатформенные npm-скрипты (`npm run setup`, `npm run verify`, …)
- не опираться на shell-специфичные команды в основном workflow: `scripts/setup.ps1` и
  `scripts/setup.sh` — точки входа для удобства, которые лишь вызывают `scripts/setup.mjs` и не
  содержат собственной логики, поэтому системы не могут разойтись
- для генерации и валидации предпочитать Node-based helper scripts
- все сторонние библиотеки, используемые в сценариях, держать локально в репозитории
- не зависеть от внешних CDN во время runtime
- тестировать только движок и инфраструктуру; страницы сценариев намеренно вариативны и не покрываются
- гейтом считать `npm run verify`, который завершается **smoke-проверкой с запуском настоящих
  серверов** (dev и статика поверх `dist/`) и проверкой их ответов: сборочный тулчейн способен сломать
  работающий стенд, пока typecheck, lint, test и build остаются зелёными
- инструменты установки не должны быть разрушающими: ставить недостающее, не вычищая существующее дерево
- данные сценариев нужно проектировать так, чтобы evaluation metadata могла расширяться без ломки маршрутов
- документировать текущее поведение отдельно от целевого и держать `docs/en` и `docs/ru` в синхроне

## Что не входит в проект

- аутентификация
- multi-user функциональность
- постоянное хранение пользовательского контента
- production backend
- серверная аналитика
